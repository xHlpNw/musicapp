# План: аватар, нарисованный в профиле

## Идея

Пользователь не загружает файл, а рисует аватар в модалке «Изменить аватар» в профиле. Рисунок с канваса отправляется на бэкенд, сохраняется и отображается как `User.avatarUrl`.

---

## 1. Бэкенд

### 1.1 Хранение файлов

- **Папка:** `{app.storage.path}/covers/avatars/` (та же база, что и обложки).
- **Имя файла:** `{userId}.png` — один файл на пользователя, перезаписывается при смене аватара.
- **Раздача:** уже работает через `GET /api/covers/avatars/{userId}.png` (CoversController отдаёт любой путь под `covers/`).
- **В БД:** `User.avatarUrl = "avatars/{userId}.png"` — фронт уже собирает URL как `/api/covers/` + `avatarUrl`.

Изменить CoversController не требуется.

### 1.2 DTO

**Файл:** `dto/auth/UpdateAvatarRequest.java`

```java
@Data
public class UpdateAvatarRequest {
    @NotBlank(message = "Image data is required")
    private String imageData; // data:image/png;base64,...
}
```

- Валидация: не пустая строка, опционально проверка префикса `data:image/png;base64,`.

### 1.3 Контроллер

**Файл:** `AuthController.java`

- Эндпоинт: `PATCH /api/auth/me/avatar`
- Тело: `@RequestBody UpdateAvatarRequest` (JSON с полем `imageData`).
- Авторизация: `@AuthenticationPrincipal SecurityUser` — только свой профиль.
- Ответ: `ResponseEntity<LoginResponse>` — обновлённый токен и пользователь (в т.ч. новый `avatarUrl`), чтобы фронт обновил состояние без повторного логина.

### 1.4 Сервис

**Файл:** `AuthService.java`

Метод `updateAvatar(Long userId, UpdateAvatarRequest request)`:

1. Декодировать base64 из `imageData` (убрать префикс `data:image/png;base64,`).
2. Проверки:
   - размер в байтах (например, макс. 200–300 KB);
   - опционально: размер изображения (макс. 256×256 или 512×512) через ImageIO или BufferedImage.
3. Создать директорию `covers/avatars/`, если её нет.
4. Записать файл `covers/avatars/{userId}.png`.
5. Обновить пользователя: `user.setAvatarUrl("avatars/" + userId + ".png")`, `userRepository.save(user)`.
6. Сгенерировать новый JWT и вернуть `LoginResponse` (как в `updateProfile`).

Зависимости: `UserRepository`, `PasswordEncoder`, `JwtUtil` уже есть; добавить инжект `@Value("${app.storage.path}")` для пути к хранилищу.

### 1.5 Безопасность

- Эндпоинт под существующим правилом `/api/**` — только для аутентифицированных.
- Меняется только свой пользователь (userId из SecurityUser).
- Ограничение размера и, при желании, разрешения картинки — защита от злоупотреблений.

### 1.6 Исключения

- Невалидный base64 или не PNG — 400 с сообщением.
- Слишком большой размер — 400.
- При сохранении файла/БД — по необходимости свои исключения или 500.

## 2. Фронтенд

### 2.1 API

**Файл:** `services/auth.service.ts`

- Метод `updateAvatar(imageData: string): Observable<LoginResponse>`.
- Запрос: `PATCH /api/auth/me`, тело `{ imageData }` **или** отдельный эндпоинт `PATCH /api/auth/me/avatar` с тем же телом (как решите на бэкенде).
- В `pipe(tap(response => this.handleAuthResponse(response)))` — после ответа обновятся токен и `currentUser$` (если бэкенд вернёт LoginResponse).

Если бэкенд сделаете `PATCH /api/auth/me/avatar`, то URL здесь: `PATCH ${this.API_URL}/me/avatar`.

### 2.2 Профиль: модалка «Изменить аватар»

**Файл:** `components/profile/profile.component.ts` (и `.html`)

Текущая модалка с полем «URL аватара» заменить на:

- **Канвас:** один `<canvas>` с фиксированным размером отображения (например 256×256 px). Внутренний размер канваса (width/height атрибуты) тоже 256×256 (или 128×128 для меньшего веса).
- **Инструменты (под или над канвасом):**
  - Цвет: `<input type="color">` и/или несколько кнопок с пресетами.
  - Толщина кисти: слайдер или кнопки (например 2, 6, 12 px).
  - Кнопка «Очистить» — очистка канваса (белый или прозрачный фон).
  - Опционально (v2): «Отменить последний штрих» (хранить историю команд или снимки состояния).
- **Поведение:**
  - При открытии модалки: инициализировать канвас (белый или прозрачный фон). Опционально: если у пользователя уже есть `avatarUrl`, загрузить изображение и отрисовать на канвасе для доработки.
  - Рисование: mousedown → mousemove → mouseup (и touch-события для мобильных), рисовать линию от предыдущей точки к текущей.
  - Кнопка «Сохранить»: `canvas.toDataURL('image/png')`, вызвать `authService.updateAvatar(dataUrl)`, в success закрыть модалку и сбросить состояние; в error показать `modalError`.

Логику рисования можно вынести в отдельный маленький сервис или класс (например `AvatarCanvasService`) или держать в компоненте.

### 2.3 Разметка и стили

**HTML:** контейнер модалки, внутри — канвас, под ним блок с инструментами (цвет, толщина, «Очистить»), внизу кнопки «Отмена» и «Сохранить».

**CSS:** канвас с границей/тенью, курсор карандаша при наведении на канвас, инструменты в одну строку или сетку.

### 2.4 Валидация на фронте

- Перед отправкой можно проверить, что на канвасе что-то нарисовано (например, не пустой по данным пикселей), и не вызывать API при пустом рисунке (или отправить и пусть бэкенд отклонит — по желанию).

## 3. Порядок реализации

| Шаг | Задача | Где |
|-----|--------|-----|
| 1 | DTO `UpdateAvatarRequest` | Backend |
| 2 | В `AuthService` метод `updateAvatar`, сохранение в `covers/avatars/{userId}.png`, обновление `User.avatarUrl` | Backend |
| 3 | В `AuthController` эндпоинт `PATCH /api/auth/me/avatar` | Backend |
| 4 | В `AuthService` (Angular) метод `updateAvatar(imageData)` | Frontend |
| 5 | В модалке профиля заменить поле URL на канвас + инструменты (цвет, толщина, «Очистить») | Frontend |
| 6 | Реализовать рисование (mouse/touch) и кнопку «Сохранить» с вызовом `updateAvatar` и обработкой ответа | Frontend |
| 7 | Стили и проверка отображения аватара после сохранения (в профиле и в шапке/сайдбаре, если есть) | Frontend |

## 4. Опционально (позже)

- Загрузка текущего аватара на канвас при открытии модалки (редактирование существующего).
- Undo последнего действия.
- Фиксированный размер изображения на бэкенде (например, ресайз до 256×256) для единообразия и экономии места.
- Поддержка touch-событий для рисования на мобильных.

После выполнения этого плана пользователь сможет нарисовать аватар в профиле, сохранить его, а бэкенд будет хранить PNG в `covers/avatars/` и отдавать по уже существующему `/api/covers/...`.

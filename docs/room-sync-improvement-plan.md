# План улучшения синхронизации комнат совместного прослушивания

Документ описывает шаги по улучшению синхронизации состояния комнаты между участниками (хост и слушатели).

---

## Текущее состояние (кратко)

- **WebSocket:** подключение к `/ws/rooms/{roomId}`, клиент только получает сообщения.
- **Рассылка состояния:** выполняется при `updateState()` и `update()` (настройки комнаты). Чат рассылается отдельно через `broadcastChatMessage()`.
- **Не рассылается:** при добавлении/удалении из очереди, при смене хоста (leave), при смене обложки комнаты (upload/delete cover).
- **Позиция:** синхронизируется только в момент действий (play/pause, seek, смена трека). Периодической подстройки позиции нет — возможен дрифт при длительном воспроизведении.
- **Безопасность WebSocket:** при подключении к комнате не проверяется, является ли пользователь участником.

---

## Приоритеты

| № | Задача | Приоритет | Сложность |
|---|--------|-----------|-----------|
| 1 | Рассылка состояния при изменении очереди | Высокий | Низкая |
| 2 | Рассылка при смене хоста и обложки | Высокий | Низкая |
| 3 | Проверка участника при подключении по WebSocket | Средний | Средняя |
| 4 | Периодическая синхронизация позиции (анти-дрифт) | Средний | Средняя |
| 5 | Надёжность: повторное подключение и начальное состояние | Низкий | Низкая |

## 1. Рассылка состояния при изменении очереди

**Цель:** при добавлении или удалении трека из очереди все участники сразу видят обновлённый список.

**Бэкенд (RoomService):**

- В методе `addToQueue()` после сохранения очереди:
  - Перезагрузить комнату с актуальной очередью (или собрать `RoomResponse` из уже загруженной сущности с обновлённым списком queue).
  - Вызвать `roomWebSocketHandler.broadcastRoomState(roomId, toDetailResponse(room))`.
- В методе `removeFromQueue()` после удаления элемента и сохранения комнаты:
  - Аналогично собрать актуальный `RoomResponse` и вызвать `broadcastRoomState(roomId, response)`.

**Важно:** для `toDetailResponse(room)` нужна актуальная коллекция `room.getQueue()`. После `roomQueueItemRepository.save(item)` и `room.getQueue().add(item)` у сущности `Room` список уже обновлён; после `removeFromQueue` — элемент удалён из коллекции и из БД. При необходимости перезагрузить комнату с `FetchType.LAZY` — вызвать `roomRepository.findById(roomId)` и строить response от перезагруженной сущности.

**Фронт:** изменений не требуется — компонент уже подписан на `state$` и при получении нового состояния обновляет `this.room` и очередь.

**Файлы:** `RoomService.java`.

## 2. Рассылка при смене хоста и при изменении обложки комнаты

**Цель:** при выходе хоста из комнаты все видят нового хоста; при смене/удалении обложки — актуальные данные.

**Бэкенд (RoomService):**

- В методе `leave()` после смены хоста (или удаления комнаты, если участников не осталось):
  - Если комната не удалена: перезагрузить комнату, собрать `RoomResponse` (toDetailResponse) и вызвать `roomWebSocketHandler.broadcastRoomState(roomId, response)`.
  - Если комната удалена: можно рассылать специальное сообщение «комната закрыта» (опционально, см. п. 5) или просто закрывать соединения — на усмотрение.
- В методах `uploadCover()` и `deleteCover()` после сохранения комнаты:
  - Вызвать `roomWebSocketHandler.broadcastRoomState(roomId, toDetailResponse(room))`.

**Фронт:** при получении обновлённого состояния уже обновляются `room` и UI (обложка, хост). Дополнительных изменений не требуется.

**Файлы:** `RoomService.java`.

## 3. Проверка участника при подключении по WebSocket

**Цель:** запретить подключение к комнате пользователям, которые не являются участниками (и не имеют права просматривать комнату).

**Проблема:** сейчас любой знающий `roomId` может открыть WebSocket и получать состояние и чат.

**Варианты реализации:**

- **A. Передача токена в query/подзаголовке**  
  Клиент подключается, например, к `wss://host/ws/rooms/123?token=...`. Сервер при `afterConnectionEstablished` проверяет токен (JWT), извлекает пользователя и проверяет членство в комнате (host или RoomMember). При неуспехе — закрыть сессию с `CloseStatus.POLICY_VIOLATION` или `CloseStatus.NOT_ACCEPTABLE`.

- **B. Отдельный эндпоинт с сессией/cookie**  
  Если WebSocket в том же домене и использует те же cookie, что и REST API, можно при первом подключении не передавать токен в URL, а на сервере получить аутентифицированного пользователя из контекста (Spring Security Session или аналог). Для стандартного Spring WebSocket без STOMP контекст пользователя может быть недоступен «из коробки» — тогда предпочтителен вариант A.

**Рекомендация:** вариант A — передача JWT в query (например `?token=...`). В `RoomWebSocketHandler.afterConnectionEstablished()`:

1. Извлечь `roomId` из path (как сейчас).
2. Извлечь токен из query, проверить JWT и получить текущего пользователя (через существующий сервис аутентификации).
3. Проверить, что пользователь — участник комнаты: хост или запись в `RoomMemberRepository` для этой комнаты.
4. При любой ошибке (нет токена, невалидный, не участник) — закрыть сессию и не добавлять в `roomSessions`.

**Файлы:** `RoomWebSocketHandler.java`, конфигурация безопасности (если нужно разрешить подключение без cookie), фронт — передача токена в URL при создании `WebSocket` в `RoomRealtimeService`.

## 4. Периодическая синхронизация позиции (анти-дрифт)

**Цель:** уменьшить расхождение позиции воспроизведения между хостом и участниками при длительном прослушивании.

**Идея:** хост периодически (например, раз в 5–15 секунд при `playing === true`) отправляет текущую позицию на сервер; сервер рассылает обновлённое состояние всем подписчикам. Участники подстраивают плеер только если разница превышает порог (например, 2–3 секунды), чтобы избежать рывков.

**Бэкенд:**

- Вариант 1 (минимальные изменения): использовать существующий `PUT /api/rooms/{id}/state` с полями `positionSeconds` и `playing: true`. Хост раз в N секунд вызывает этот эндпоинт; сервер обновляет `positionSeconds` и вызывает `broadcastRoomState`. Минус: лишние записи в БД и полный объект состояния в ответе/рассылке.
- Вариант 2: ввести отдельный тип сообщения по WebSocket «только позиция» (см. ниже) и не сохранять её в БД при каждом тике — только рассылать. Тогда периодические обновления не засоряют БД.

**Фронт (хост):**

- В компоненте комнаты (или в отдельном сервисе) при `room.playing === true` и когда пользователь — хост запустить интервал (например, каждые 10 секунд):
  - Считать текущее время плеера `playerService.getCurrentTime()`.
  - Вызвать `roomService.updateState(roomId, { positionSeconds, playing: true, currentTrackId, queueItemId })` или отдельный лёгкий эндпоинт «sync position».
- При паузе или смене трека интервал останавливать.

**Фронт (участники):**

- При получении состояния по WebSocket с типом «полное состояние» применять текущую логику `syncPlayerToRoom`.
- При получении сообщения «только позиция» (если введёте такой тип): если разница между текущим временем плеера и `positionSeconds` по модулю больше порога (например, 2 с) — выполнить `requestSeek(positionSeconds)`; иначе не трогать плеер.

**Опционально — отдельный тип сообщения «position»:**

- Сервер: новый метод, например `broadcastPosition(Long roomId, double positionSeconds)`.
- Формат: `{ "type": "position", "positionSeconds": 123.45 }`.
- Рассылать только при периодическом обновлении от хоста; при play/pause/seek по-прежнему рассылать полное состояние.
- На клиенте в `room-realtime.service.ts` разбирать сообщения: при `type === 'position'` эмитить в отдельный поток или вызывать колбэк; при полном состоянии — как сейчас в `state$`.

**Файлы:** бэкенд — `RoomService` (или отдельный эндпоинт), `RoomWebSocketHandler` (при введении `broadcastPosition`); фронт — `RoomDetailComponent` или сервис таймера, `RoomRealtimeService`, `room.service.ts` (при отдельном эндпоинте).

## 5. Надёжность: повторное подключение и начальное состояние

**Цель:** при обрыве WebSocket участник быстрее получает актуальное состояние; при входе в комнату не ждать первого события по WebSocket.

**Повторное подключение:**

- На фронте в `RoomRealtimeService` при `onclose`/`onerror` не вызывать немедленный reconnect (чтобы не создать цикл при 403/404), но при переходе обратно на страницу комнаты вызывать `connect(roomId)` заново — это уже делается при `loadRoom`. При желании можно добавить ограниченные попытки переподключения с задержкой (exponential backoff), если соединение закрылось не по коду 4xxx.

**Начальное состояние после подключения:**

- После успешного открытия WebSocket на клиенте один раз запросить текущее состояние комнаты по REST (`GET /api/rooms/{id}`) и применить `syncPlayerToRoom(response)`. Так участник сразу получает актуальные трек и позицию, не дожидаясь первого push (который может прийти только при следующем действии хоста).
- Альтернатива: сервер при подключении клиента к комнате сразу отправляет ему одно сообщение с текущим состоянием (при наличии доступа к RoomService в Handler). Тогда отдельный REST-запрос не нужен.

**Сообщение «комната закрыта»:**

- При удалении комнаты (последний участник вышел) можно рассылать сообщение `{ "type": "roomClosed", "roomId": ... }`. Клиент при получении закрывает WebSocket, показывает уведомление и перенаправляет на список комнат. Реализация опциональна.

**Файлы:** фронт — `RoomRealtimeService`, `RoomDetailComponent`; бэкенд — при желании `RoomWebSocketHandler` (отправка состояния при подключении, рассылка roomClosed).

## Порядок внедрения (рекомендуемый)

1. **П. 1 и 2** — рассылка при изменении очереди, смене хоста и обложки. Быстро устраняет основные расхождения в данных.
2. **П. 5 (начальное состояние)** — один запрос состояния по REST после открытия WebSocket (или отправка состояния с сервера при подключении). Улучшает UX при входе в комнату.
3. **П. 3** — проверка участника при подключении по WebSocket. Повышает безопасность.
4. **П. 4** — периодическая синхронизация позиции. Улучшает качество синхронизации при длительном прослушивании.
5. **П. 5 (повторное подключение / roomClosed)** — по необходимости.

После выполнения пунктов 1–2 синхронизация очереди и метаданных комнаты будет существенно лучше; пункты 3–5 повысят безопасность и плавность совместного прослушивания.
